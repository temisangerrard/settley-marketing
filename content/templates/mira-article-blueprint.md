# Mira Article Blueprint

Use this before drafting any Settley article. This is not optional guidance. It is the fixed editorial structure for pages that should feel premium, trustworthy, and conversion-ready.

## Purpose

This template exists to stop three recurring failures:

1. Template leakage: duplicated labels, repeated titles, stray placeholders, broken footer fragments.
2. Fake interactivity: pages that promise an interactive experience but mostly read like a long wall of text.
3. Flat hierarchy: too many sections with the same visual weight, no clear narrative, and a weak CTA.

## Non-Negotiables

- One title only.
- One category label only.
- One primary CTA direction only.
- One hero promise only.
- Maximum two interactive embeds on a page.
- If an interactive element exists, it must deliver value within 5 seconds on mobile.
- No generic stat row unless each stat changes the reader's decision.
- No paragraph longer than 4 lines at desktop width.
- No section that exists only because "most blog posts have one."

## House Style

- Tone: precise, calm, intelligent, not breathless.
- Brand signal: premium education, not startup template, not fintech hype.
- Reader promise: "I will understand this clearly and know what to do next."
- Visual rule: every section must earn its space by changing pace, medium, or level of specificity.

## Fixed Article Structure

### 1. Metadata

Fill these first:

```text
META_TITLE:
META_DESCRIPTION:
ARTICLE_TITLE:
ARTICLE_SUBTITLE:
ARTICLE_CATEGORY:
PUBLISH_DATE:
UPDATED_DATE:
READ_TIME:
PRIMARY_CTA_URL:
PRIMARY_CTA_LABEL:
```

### 2. Hero

Required:

- Category label
- H1
- One-sentence subtitle
- Meta row
- One sharp opening promise

Rules:

- Do not repeat the category in the breadcrumb and then again as a big visual event.
- Do not say "interactive" unless the interaction is visible above the fold.
- Do not stack multiple promises in the subtitle.

### 3. Opening Proof Block

Pick one:

- Key takeaways box
- Decision snapshot
- Myth vs reality card strip

Rules:

- Use 3 to 5 points only.
- Each point must be decision-relevant, not definitional filler.

### 4. Core Visual Moment

Pick one primary device:

- Interactive diagram
- Calculator
- Comparison card stack
- Annotated flow

Rules:

- This is the main event of the page.
- It must appear early.
- It must be understandable without reading the full article.
- Add one sentence below it explaining what the reader should notice.

### 5. Explanation Section

Use 2 to 4 short sections:

- What it is
- How it works
- What the reader actually owns / gets / risks
- Why this matters now

Rules:

- Prefer cards, comparisons, and short paragraphs over long sequential prose.
- If a table is used, the left column must create contrast, not repetition.

### 6. Constraint or Risk Section

Required for investment, property, regulation, or returns-related content.

Use:

- Risk callout
- Constraint list
- "Where this breaks down" section

Rules:

- This should feel sober and visually distinct.
- No soft-pedalling risk language.

### 7. Action Section

Show what the reader can actually do now:

- Explore a deal
- Join the waitlist
- Compare routes
- Read a next guide

Rules:

- Make the transition explicit: "If this matches what you want, do X."
- Do not bury the CTA after endless FAQ gravity.

### 8. FAQ or Sources

Only include FAQ if the query intent genuinely needs it.

Rules:

- 3 to 5 questions max.
- If the FAQ is weak, remove it and strengthen the action section instead.

### 9. Final CTA

Required:

- One headline
- One support sentence
- One button

Rules:

- No second competing CTA block elsewhere on the page.
- CTA copy should complete the article's promise.

## Design Rules

### Hierarchy

- Large moments: hero, main visual, CTA.
- Medium moments: section heads, callouts, comparison blocks.
- Small moments: metadata, source notes, disclaimers.

If everything feels medium, the page will feel generic.

### Images

- Check `/Users/temisan/Downloads/settley-marketing/assets/images` before sourcing or generating article visuals.
- Prefer images from that folder when they fit the article instead of reaching for external stock by default.
- If a local image is used, make it earn its place: support the argument, improve scannability, or break up dense sections.
- If none of the local images fit the article, then use another source deliberately rather than ignoring the folder by habit.

### Interactivity

Interactive blocks must:

- Have one obvious control or one obvious click target.
- Explain what changes when interacted with.
- Work on mobile without precision tapping.
- Avoid looking like an embedded prototype dumped into an article.

### Tables

Use tables only when side-by-side contrast matters.

Do not use tables for:

- obvious definitions
- long sentences
- copy that belongs in cards

### Stats

Use a stat row only when the numbers are concrete proof.

Bad:

- "4 steps"
- "Monthly"
- "SPV"

Good:

- "Min £500 entry"
- "6.2% net yield example"
- "3-year avg void period: X days"

### Copy Pacing

Alternate:

- explanation
- proof
- visual
- implication

Do not stack six prose sections in a row.

## Hard Bans

- Duplicate article title
- Duplicate category label
- "Interactive" in three different places
- More than one hero summary block
- Generic stat filler
- Broken placeholder artifacts
- Internal instruction copy leaking into reader-facing prose
- Meta lines that talk to Mira instead of the reader
- Phrases like "use this as the house reference", "future published articles", or "this file is a template"
- Footer junk
- Two embeds back to back without interpretation
- CTA only at the very bottom after excessive legal copy

## Template Discipline

The blueprint is for Mira, not for the reader.

- Never publish instructional text that explains how to write the article.
- Never leave template coaching, QA reminders, or process notes inside CTA blocks, FAQ answers, captions, or disclaimers.
- Convert every section into reader-facing copy before publishing. If a sentence would only make sense to an editor, remove or rewrite it.
- Disclaimers should cover legal/financial caution only. They should not mention templates, publishing workflow, or future review steps.
- Before shipping, do one explicit leakage pass: scan for words like `template`, `canonical`, `reference`, `future article`, `publish target`, `house standard`, and `Mira`.

## Pre-Publish Review

Before publishing, verify:

- The page can be understood by skimming headings alone.
- The strongest visual appears before the reader has to commit.
- The CTA feels earned, not bolted on.
- Mobile view still makes the core interactive block obvious.
- Nothing on the page looks auto-generated or unreviewed.
- No line reads like an instruction to the writer or a note from the template.
